什么是Node.js？

发表于 2018/04/06 22:47:59   |   分类于 :  分享   |   访客 : 89   评论 : 0

Node.js的。这是“你是否足够酷，可以使用我？”编程语言，API和工具包的最新版本。从这个意义上说，它完全符合Rails，Ajax和Hadoop的传统，甚至在某种程度上也支持iPhone编程和HTML5。参加一个大型的技术会议，你几乎可以肯定会找到关于Node.js的一些会谈，虽然大多数会飞到普通凡人程序员的头上。

深入挖掘一下，你会听到Node.js（或者，因为它被很多人简单地称为“Node”）是一个用于JavaScript的服务器端解决方案，特别是用于接收和响应HTTP请求。如果这并不能完全令你大开眼界，那么在讨论端口，套接字和线程时，谈话就会升温，你会倾向于釉下去。这真的是JavaScript吗？事实上，为什么世界上有人想在浏览器外部运行JavaScript，更不用说服务器了？

好消息是你正在听到（并思考）正确的事情。节点确实关心网络编程和服务器端请求/响应处理。坏消息是，就像之前的Rails，Ajax和Hadoop一样，有很少的清晰信息可用。随着时间的推移 - 现在已经有了这些其他已经成熟的“酷”框架 - 但是为什么要等待一本书或教程，以便您今天能够使用Node，并且显着提高代码的可维护性，甚至是你为程序员带来的便利？

获取O'Reilly的每周编程时事通讯

对那里的Node专家提出警告

节点就像大多数人所熟悉的大多数技术一样，但是对于有经验的人来说却是一种老式的帽子：它对大多数人来说是不透明和奇怪的，但对于一个小团体来说却完全可用。结果是，如果你从未使用过Node，那么你需要从一些非常基本的服务器端脚本开始。花点时间确保你知道发生了什么，因为虽然这是JavaScript，但它不像你习惯的客户端JavaScript那样操作。事实上，你将不得不围绕事件循环和等待，甚至是一些网络理论来扭转你的JavaScript大脑。

不幸的是，这意味着如果你一直在和Node一起工作和玩一两年，那么这篇文章的大部分内容似乎都是行人和过分简单的。你会寻找诸如在客户端上使用Node的东西，或者关于偶数I / O和反应堆模式的重大理论讨论。现实情况是，尽管这很有趣 - 并且将Node推进到一些非常史诗级的地位 - 但对于刚开始的人来说，这是不可理解的。鉴于这种情况，也许你应该通过这片到您的同事谁不知道节点，然后当他们购买到节点的用处，开始给他们带来沿上更先进的节点使用情况。

节点：几个基本的例子

首先要做的事情是：你需要认识到Node是用来运行独立的JavaScript程序的。这不是一段HTML并在浏览器中运行的文件。它是一个坐落在文件系统上的文件，由程序执行，作为守护进程运行，监听特定的端口。

跳过你好的世界

这里的典型例子是“Hello World”，在Node网站上详细介绍。几乎每个人都从Hello World开始，所以请自行检查一下，直接跳到更有趣的事情：可以发送静态文件的服务器，而不仅仅是一行文本：


	var sys = require("sys"),
	    http = require("http"),
	    url = require("url"),
	    path = require("path"),
	    fs = require("fs");

	http.createServer(function(request, response) {
	    var uri = url.parse(request.url).pathname;
	    var filename = path.join(process.cwd(), uri);
	    path.exists(filename, function(exists) {
	        if(!exists) {
	            response.writeHead(404, {"Content-Type": "text/plain"});
	            response.end("404 Not Foundn");
	            return;
	        }

	        fs.readFile(filename, "binary", function(err, file) {
	            if(err) {
	                response.writeHead(500, {"Content-Type": "text/plain"});
	                response.end(err + "n");
	                return;
	            }

	            response.writeHead(200);
	            response.end(file, "binary");
	        });
	    });
	}).listen(8080);

	console.log("Server running at http://localhost:8080/");

快速的逐行引物

关于Node的讨论比运行服务器的实际代码还要多。不过，在继续之前，值得通过NodeFileServer.js进行一次快速的巡航。再看看代码：


var http = require('http');
http.createServer(function (req, res) {
  res.writeHead(200, {'Content-Type': 'text/plain'});
  res.end('Hello Worldn');
}).listen(1337, "127.0.0.1");
console.log('Server running at http://127.0.0.1:1337/');

首先，你有一个叫做函数的调用。程序员一直在使用它。实际上，您可以在一些关于JavaScript模块化的讨论中提到这一点，以及与CommonJS的密切关系，以及来自2009年的O'Reilly作者David Flanagan的非常酷的实现。换句话说，对您而言可能是新的，但它不是这是一个未经测试的，粗心的节点代码。它是使用模块化JavaScript的核心，也是Node极具优势的东西。

然后，生成的变量用于创建服务器。该服务器交给一个功能块，以便在联系时运行。这个特定的函数完全忽略了请求，只是写出一个响应，简单地说，“Hello Worldn”。非常简单的东西。

实际上，这为节点使用规定了标准模式：

  1. 定义交互的类型并获取用于处理该交互的变量（通过）。
  2. 创建一个新的服务器（通过）。
  3. 手中的服务器处理请求的功能。
    • 请求处理函数应该包含一个请求...
    • ...和回应。
  4. 通知服务器开始处理特定端口和IP（通过）的请求。

迷失在翻译

尽管你可以轻松获得一个用JavaScript编写的服务器（不管实际的代码运行工具是C还是其他），但仍然有问题：你应该用JavaScript编写一个服务器吗？要真正解决这个问题的答案，请考虑一个非常典型的用例。

JSON往返

您有一个典型的Web应用程序，带CSS样式的HTML前端，以及用于验证和与服务器通信的JavaScript。而且，由于您在交互式网络上，因此您使用的是Ajax，而不是单纯依靠表单的POST来从服务器获取数据。如果这是你，那么你也可能对JSON感到满意，因为这几乎是事实上通过电线发送数据的手段。

所以你有一个Ajax请求，例如说：“给我更多关于某个在线拍卖站点上某种特定吉他的信息”。该请求会通过网络传播到在某个服务器上运行的PHP程序。PHP服务器必须将大量信息发送回JavaScript请求者，并且需要以JavaScript可以解压缩的格式发送该信息。因此，返回信息被捆绑到一个数组中，然后可以将其转换为JSON，如下所示：


$itemGuitar = array(
  'id' => 'itemGuitar',
  'description' => 'Pete Townshend once played this guitar while his own axe ' .
                    was in the shop having bits of drumkit removed from it.',
  'price' => 5695.99,
  'urls' => array('http://www.thewho.com', 'http://en.wikipedia.com/wiki/Pete_Townshend')
);

$output = json_encode($itemGuitar);
print($output);

回到客户端，JavaScript获取了这些信息，这些信息由于JSON和传输而略有变化。客户基本上得到这样的东西：


{
  "id": "itemGuitar",
  "description": "Pete Townshend once played this guitar...",
  "price": 5695.99,
  "urls": ["http://www.thewho.com", "http://en.wikipedia.com/wiki/Pete_Townshend"]
}

这是相当标准的票价。然后，很容易将此文本“东西”转换为JavaScript中的对象。你只需打电话，像这样：


var itemDetails = eval('(' + jsonDataString + ')');

结果是一个很好的JavaScript对象，其属性与JSON数组结构类似。当然，由于通常是从服务器返回的，因此您更可能看到如下代码：


var itemDetails = eval('(' + request.responseText + ')');

这是典型的JSON往返行程。但是这里存在问题......大的问题。

微妙和微妙破坏固体代码

首先，这种代码严重依赖于翻译器，这是一个主要问题。在这种情况下，翻译器是JSON解释器和相关的代码，实际上有两个依赖关系：用于Java的JSON解释器，其形式为响应文本，以及PHP的JSON解释器。从PHP 5.2.0开始，该解释器包含在PHP中，但它仍然是一个外部依赖项，与PHP的核心相分离。

现在，这不是关于翻译本身的咆哮。没有任何迹象表明在将一个“l”变成一个“i”或者某个数组中的项目1并将其报告为数组中的项目2时存在问题。在发布JSON工具之前会进行很多测试，以确保报告的内容是正确的，并且可以实现从客户端到服务器的准确回滚。很多很多的测试都涉及到......

这实际上是一个问题。

工具包中的JavaScript和PHP（以及C和Lisp，Clojure和Eiffel等）的依赖关系是一个巨大的问题，这个依赖关系在所有的JSON工具包中都有很多不同的语言。换句话说，这个问题是不是transla 重刑，但transla TOR。虽然编程语言进展缓慢，但这些语言应用的用途正在迅速增长。其结果是，JSON被用于复杂的领域，这些领域甚至在几个月前根本不存在或者没有改变。每次新的迭代 - 递归的每个新深度和数据类型的组合 - 可能发现翻译者不支持的区域。

这本身并不坏。事实上，它认为JSON的流行使它不断得到新的使用。但是，随着“新”的出现，“它是否支持新的？”所以JSON必须不时发展，这意味着在大量平台上进行测试，重新测试和发布。程序员您可能需要重新排列数据; 或等待发布以支持您的需求; 或者自己破解JSON。再次，其中许多只是所谓的编程成本。

但想象一下，你可以把翻译 - 因此翻译 - 完全抛弃。想象一下，你可以写，而不是JSON的绊倒，但JavaScript的首尾。

这是Node的承诺。您刚刚阅读的所有文本 - 关于包括JSON在5.2.0中的PHP，但不包括之前的内容，关于数组成为对象，关于以新方式配置数据以及从JSON需要新东西的所有文本 - 当JavaScript发送数据时，这些都会消失并接收并响应该数据。

JavaScript中的eval（）是（可能）恶魔

就好像没有足够的理由认真对待Node一样，存在一个在字符串上运行的讨厌问题。很久以前就被认为 是危险的东西。它运行的代码只能看作文本数据; 它相当于“在这里输入SQL来运行你的SQL”未经验证的文本框，可以打开SQL注入和恶意的意图。很可能每一次都是通过一根绳子传到中西部地区的一只小狗身上，而东海岸的一位母亲却把她的脚趾和诅咒残留下来。这是不稳定的。网上有很多需要阅读的内容，这里不值一提。只需Google “评估JavaScript恶意”或“评估JavaScript注入”就可以很好地体验这些问题。

但是，没有任何上下文的节点不允许你避免，所以有可能仍然在发抖的小狗。但是，按照预期使用的Node 绝对可以解决典型问题。节点通常被称为是平滑的JavaScript或者是偶数的I / O，而这个小小的词 - “平衡” - 非常重要。但是要了解事件的真正含义，以及为什么它能让你摆脱危险，您不仅要了解JSON如何在应用程序中被一次性访问，而且Web应用程序的结构如何通常是架构的。

今天的网络是一个大事件网站

典型的网络形式是“大事件”提交者。换句话说，大量数据输入和选择发生 - 用户填写文本框，从组合框中选择选项，从列表中选择项目等等 - 然后将所有这些信息提交给服务器。从编程的角度来看，有一个“大事件”：通常通过POST提交所有表单数据。这几乎是网络运行的前期Ajax。

一次发送大量数据

使用Ajax，有更多一点的什么所谓的事件触发编程。有更多的事件触发与服务器的交互。经典案例是输入一个邮政编码，然后调用服务器来获取城市和州。通过Ajax和XmlHttpRequest，大量的数据不必一下子就变成一团糟并投入服务器。然而，这并没有改变网络仍然是一个大事件的现实。Ajax用于实现有趣的视觉效果，进行快速验证以及在不离开页面的情况下提交表单，而不是创建真正的均衡网页。所以，即使表单没有提交POST信息，Ajax请求也是如此。

老实说，这只是部分缺乏创造力的Ajax程序员的错。每次发送请求时 - 无论多小 - 都会发生大量网络流量。服务器必须响应该请求，通常在其自己的线程中使用新进程。因此，如果你真的转向了一个平衡的模型，那里你可能有10或15个单独的微型请求从一个页面到服务器，你将有10或15个线程（可能更少，这取决于线程如何汇集以及它们在服务器上回收的速度）开始启动。现在再乘以1,000或10,000或1,000,000份给定页面，并且你可能会出现混乱。网络放缓。系统崩溃。

结果是，在大多数情况下，Web 至少需要一个中等事件地点。这一让步的结果是，服务器端程序不会针对非常小且集中的请求发送微小响应。他们发送多位数据，这需要JSON，然后你又回到了问题。问题是肯定的，但问题也是 - 从某个角度来看，至少 - 网页的性质以及网页和响应该请求的服务器端程序之间的线程和HTTP流量。

（你们中有些更高级的JavaScript人士在此尖叫，因为你知道比使用更好，相反，你使用的是类似的东西，而且还有一些引人注目的用法，这些东西值得尖叫不过，只要看看Stack Overflow等站点上有多少个问题，你就会意识到大多数人不能正确或安全地使用，这是一个问题，因为有许多中间程序员只是不知道的问题。）

随时发送少量数据

节点为聚会带来了一种不同的方式：它试图将你和你的web应用转移到一个平衡的模型，或者如果你喜欢，就是一个“小事件”模型。换句话说，与其发送少量的数据请求，你应该发送大量的请求，大量的事件，微小的数据，或需要响应的请求只有一点点的数据。在某些情况下，你几乎不得不回忆你的GUI编程。（所有的Java Swing人们终于可以使用他们的被压抑的图形用户界面知识）。因此，用户输入他们的名字和姓氏，并且当他们移动到下一个框时，请求已经请求对照现有名称对该名称进行验证。邮政编码，地址和电话号码也是如此。有不断发出的请求和响应流，与页面上几乎所有可以想象的事件相关联。

那有什么区别？为什么Node有可能这样做，并且在线程中存在的问题并不是相同的问题？那么，不，他们不是。Node自己的网站解释他们的哲学是最好的：

Node的目标是提供一种构建可扩展网络程序的简单方法。在“ hello world ”web服务器示例中...许多客户端连接可以同时处理。节点告诉操作系统（通过epoll，kqueue，/ dev / poll或select）在新建连接时应该通知操作系统，然后进入睡眠状态。如果有人新连接，则执行回调。每个连接只有一个小堆分配。

节点没有块，没有线程竞争相同的资源（节点很高兴让事情发生，但它们发生），没有什么必须根据请求启动。节点只是坐在等待中（毫不夸张地说，未使用的节点响应者正在睡觉）。当请求进入时，它被处理。这导致代码非常快，没有超级程序员编写服务器端行为。

是的，混乱会随之而来

值得指出的是，这个模型确实允许任何非阻塞系统允许进入的所有问题：一个进程（而不是线程）写入数据存储，而另一个进程则捕获刚刚失效的数据; 侵入交易的金额; 等等。但要意识到，Web表单上的大部分基于事件的编程都是只读的！你多久实际上在一个微型请求中修改数据？非常稀有。相反，有一个持续的验证，数据查询和查询正在进行。在这些情况下，最好只是解决请求。数据库本身可能会添加一些锁定，但总的来说，好的数据库将比服务器端代码更有效地完成此操作; 而且他们肯定会处理好事情，而不是操作系统会在一个通用的“线上响应”过程中启动和停止线程。

另外，Node有计划允许进程分叉，而HTML5 Web Workers API是可能使这个特性发挥作用的引擎。尽管如此，如果您转向Web应用程序的平衡模型，您可能会遇到一个问题，您可能需要在100个以内的情况中使用不到一个的线程。尽管如此，这些更改最适合您考虑Web应用程序的方式，以及您从服务器发送和接收数据的频率，而不是Node如何工作。

在正确的时间正确的地方

这里还有另一种Web模式，它可能比使用Node还是更重要，以及Web应用程序的使用情况。这仅仅是：针对不同的问题使用不同的解决方案。更好的是，针对特定问题使用正确的解决方案，而不管这是否是您用于解决所有其他问题的解决方案。

熟悉的惯性

不仅仅是网页设计，而是所有的编程都有一定的惯性。这种惯性可以用这样的公理来表述：你学习，使用越多，擅长某种方法或技术或语言，就越有可能广泛使用这种方法/技术/语言。这是在你深入挖掘之前听起来很好的原则之一。是的，很好地学习一门语言或工具包，并广泛使用它。但是这种惯性往往会让你使用一种工具，因为你知道它，而不是因为它是正确的工具。

看看已经讨论过的Ajax。最初，Ajax提供了一种可靠的方法，可以将不需要表单提交的快速请求发送到服务器。现在它已成为所有表单提交的直接替代品。这需要一种技术，学习它，应用它，然后最终过度使用它。表单提交仍然是一个坚实的地方 - 当表单需要提交时！尽管听起来很简单，但有成千上万的Web应用程序往往使用Ajax提交表单，这只是因为主要的Web开发人员使用了Ajax。

同样，可能会对Node感到兴奋 - 可能是因为你购买了所有精彩且明智的观察结果 - 然后在任何地方使用。突然间，你用Node替换了所有的PHP和Perl后端。结果？一团糟。事实上，您将被迫使用几种Web表单来完成Node 不适用的功能：通过Node将大量数据提交给服务器上的JavaScript，并强制该JavaScript返回一大块JSON被解析或编辑，或者发回一个完整的HTML页面或HTTP重定向。

但这根本不是Node最擅长的。在微型请求中很棒; 在偶数I / O。使用节点在网页和服务器之间进行快速通信。使用表单提交将大块数据发送到服务器。使用PHP和Perl来完成沉重的数据库并生成动态HTML页面。使用Node为服务器端JavaScript提供运行和处理小型请求的方法。抛出Rails，Spring和Servlet以及其他你需要的东西。但是要根据你正在解决的问题做出决定，而不是你当时最了解的事情。

Node简单的承诺

有最后一个值得做的事。当您采用这种广泛的编程方法时，您经常会发现您不必深入研究每个使用的工具包，API和框架。通过使用自己最擅长的工具，您不需要用锤子装订或用钻头进行测量。将工具用于其预期用途通常意味着您可以更多地使用核心功能。所以当你创建多​​面手时 - 程序员知道很多东西 - 你也减少了对专家的需求 - 程序员真的很好地知道一两件事情。当然，每个尖尖的老板也意识到这些专家真的很贵，很难找到。

学习节点可能需要一点努力，但它会取得成功。为什么？因为您为您的Web应用程序问题提供了解决方案，只需JavaScript即可解决问题。这意味着您现有的JavaScript专业知识发挥作用。当你需要使用PHP或Perl时 - 因为它是解决特定问题的正确解决方案 - 你不需要PHP或Perl专家。您需要了解基础知识，并且在问题需要扩展时可以扩展这些需求。拉伸应对新问题的要求，而不是简单地拉伸穷人的解决方案。

你面临的最大挑战是不断移动到由小块组成的网络，更频繁的讨论，以及看起来像令人眼花缭乱的技术组合的组合。但是，采用100种技术的核心特性总是会比单独采用100％技术并试图解决100个问题更好地为您服务。节点和偶数I / O不是每个问题的解决方案，但它肯定是一些重要问题的解决方案。
